Ранее в этой статье уже шла речь про хранение текстов формата .md в GitHub, что даёт универсальность в работе над контентом через Git. С мая 2025 года доступен нативный коннектор для подключения репозитория к ChatGPT через OAuth. 

![[GH+CGPT-0.png]]

В настройках ChatGPT есть пункт с "Приложения и коннекторы", где можно указать до 5 репозиториев, к которым у модели будет доступ.

![[GH+CGPT-1.png]]

Стоит отметить, что от качества организации репозитория зависит итоговый результат. Чтобы модель обращалась к конкретной информации, нужно включить эту опцию перед отправкой запроса и детально проработать промпт.​ 

![[GH+CGPT-2.png]]

В контексте нашей экосистемы это ещё одно связующее звено, благодаря которому у языковой модели растёт понимание контекста запроса и таким образом ускоряется поиск необходимой информации. Прежние тексты становятся основой для новых.

Как бы хорошо ни звучала эта идея в теории, на практике загрузить целиком весь репозиторий в диалог не получится. Файлы обрабатываются через механизм семантического поиска (Retrieval-Augmented Generation, RAG). Так, ChatGPT преобразует запрос в векторное представление, находит релевантные данные через интеллектуальный анализ базы знаний, после чего выбранные фрагменты попадают в выдачу с указанием источника. Для большей полноты ответа есть режим «Deep Research», в котором можно одновременно работать с несколькими документами сразу.​

Факт — результат ответа существенно зависит от контекста. Модель не просто даёт общий ответ, а создаёт уникальный текст, идеально подходящий для проекта. Если в репозитории есть такие полезные вещи как чёткая конвенция об именовании и рабочей терминологии, стайлгайд, задокументированный ToV, то на их основе ChatGPT будет генерировать так, чтобы адекватно вписываться в существующий контент.

При отправке запроса типа *"Найди в репозитории информацию о методе работы с контентом и объясни, как он работает в контексте создаваемой статьи"*, ChatGPT использует семантический поиск для нахождения файлов, связанных с регистрацией, затем анализирует их содержимое и интегрирует эту информацию в ответ с указанием источников из репозитория.

В предыдущих главах мы рассказали о возможностях Cursor IDE для авторов. Сам редактор и ассистент внутри него уже видят файлы, которые хранятся локально, и именно они подпитывают автодополнение в процессе работы. Подключение GitHub полезно тем, что позволяет языковой модели видеть историю коммитов, pull request-ов, issue и веткам, которые ещё не синхронизированы локально.

Именно поэтому при запросах вроде *@folder:docs* можно быть уверенным, что модель учитывает актуальное состояние удалённой ветки, даже если она расходится с локальной копией. В результате команда получает единый контекст, а подсказки ChatGPT учитывают не только локальные черновики, но и свежие изменения коллег.

Ключевой момент для команды: Cursor сохраняет контекст только для текущей сессии, и при перезапуске редактора проектная информация может быть утеряна. Важно, чтобы каждый участник запустил сессию с нужным репозиторием, проверил корректность конфигурации моделей и помнил, что открытые файлы сохраняют контекст, но общий контекст проекта может сброситься, если достигнут лимит.

Так GitHub становится централизованным хранилищем знаний, повышая эффективность контентной команды через интерфейсы с доступом к ИИ-функционалу.
